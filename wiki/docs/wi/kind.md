# Kind

kind — это инструмент для запуска локальных кластеров Kubernetes с использованием контейнеров Docker в качестве «узлов». kind изначально был разработан для тестирования самого Kubernetes, но может использоваться для локальной разработки или в CI.

## Установка в Альт (Пересборка)

Итак, по состоянию на 21.07.26 Kind собран только в Сизифе, поэтому мы просто пересоберём у себя на хосте из srpm.

Устанавливаем зависимости:
```bash
apt-get install rpm-macros-golang rpm-build-golang golang
```

Скачиваем пакет:
```bash
curl https://git.altlinux.org/tasks/archive/done/_388/398111/build/100/x86_64/srpm/kind-0.30.0-alt1.src.rpm
```

Пересобираем:
```bash
rpmbuild --rebuild kind-0.30.0-alt1.src.rpm
```

Устанавливаем:
```bash
apt-get install ~/RPM/RPMS/x86_64/kind-0.30.0-alt1.x86_64.rpm
```

## Обнаружение дополнительных параметров команд

В kind встроена справка для всех команд и подкоманд.
Вы можете изучить доступные флаги и подробности использования, выполнив:

```bash
kind help
kind --help
kind <команда> --help
```

Это относится и к таким командам, как `kind load image-archive`.

## Создание кластера

Создать кластер Kubernetes так же просто, как выполнить:

```bash
kind create cluster
```

Это загрузит кластер Kubernetes, используя предварительно собранный образ узла. Готовые образы размещены на `kindest/node`, но чтобы найти образы, подходящие для данного релиза, вам следует обратиться к примечаниям к выпуску вашей версии kind (проверьте через `kind version`) – там вы найдёте полный список образов, созданных для этого релиза.

Чтобы указать другой образ, используйте флаг `--image`:

```bash
kind create cluster --image=...
```

Использование другого образа позволяет изменить версию Kubernetes создаваемого кластера.

Если вы хотите собрать образ узла самостоятельно с пользовательской версией, обратитесь к разделу [Сборка образов](#сборка-образов).

По умолчанию кластер будет назван `kind`. Используйте флаг `--name`, чтобы задать другое имя контекста.

Если вы хотите, чтобы команда `create cluster` блокировала выполнение до достижения плоскостью управления состояния готовности, используйте флаг `--wait` и укажите таймаут. Для `--wait` необходимо указать единицы времени ожидания. Например, чтобы ждать 30 секунд, выполните `--wait 30s`, для 5 минут – `--wait 5m` и т.д.

Дополнительные возможности можно узнать с помощью:

```bash
kind create cluster --help
```

kind может автоматически определить установленные docker, podman или nerdctl и выбрать доступный. Если вы хотите отключить автоопределение, используйте переменную окружения `KIND_EXPERIMENTAL_PROVIDER=docker`, `KIND_EXPERIMENTAL_PROVIDER=podman` или `KIND_EXPERIMENTAL_PROVIDER=nerdctl`, чтобы выбрать среду выполнения.

> **ПРИМЕЧАНИЕ:** podman и nerdctl по умолчанию работают в режиме rootless. Для полноценной работы кластеров KIND требуется дополнительная настройка.

## Взаимодействие с вашим кластером

После создания кластера вы можете использовать `kubectl` для взаимодействия с ним, используя файл конфигурации, сгенерированный kind.

По умолчанию конфигурация доступа к кластеру сохраняется в `${HOME}/.kube/config`, если не установлена переменная окружения `$KUBECONFIG`.

Если переменная `$KUBECONFIG` установлена, она используется как список путей (с разделителями, принятыми в вашей системе). Эти пути объединяются. При изменении значения оно изменяется в том файле, который определяет соответствующую секцию. При создании значения оно создаётся в первом существующем файле. Если ни один файл в цепочке не существует, то создаётся последний файл в списке.

Вы можете использовать флаг `--kubeconfig` при создании кластера – тогда будет загружен только этот файл. Флаг может быть указан только один раз, и объединение не выполняется.

Чтобы увидеть все созданные вами кластеры, используйте команду `get clusters`.

Например, создадим два кластера:

```bash
kind create cluster          # Имя контекста по умолчанию — `kind`.
...
kind create cluster --name kind-2
```

При выводе списка кластеров kind вы увидите примерно следующее:

```bash
kind get clusters
kind
kind-2
```

Чтобы взаимодействовать с конкретным кластером, достаточно указать имя кластера как контекст в `kubectl`:

```bash
kubectl cluster-info --context kind-kind
kubectl cluster-info --context kind-kind-2
```

## Удаление кластера

Если вы создали кластер с помощью `kind create cluster`, удаление выполняется так же просто:

```bash
kind delete cluster
```

Если флаг `--name` не указан, kind использует имя кластера по умолчанию `kind` и удаляет этот кластер.

> **Примечание:** По замыслу, запрос на удаление несуществующего кластера не возвращает ошибку. Это сделано намеренно, чтобы обеспечить идемпотентный способ очистки ресурсов.

## Загрузка образа в кластер

Вы можете загрузить один или несколько образов в ваш кластер kind:

```bash
kind load docker-image my-app:latest
kind load docker-image my-app:latest my-db:latest my-cache:latest
```

> **Примечание:** Если вы используете именованный кластер, нужно указать его имя:
> ```bash
> kind load docker-image my-app:latest --name test-cluster
> ```

Кроме того, можно загружать архивы образов с помощью:

```bash
kind load image-archive /my-image-archive.tar
```

Это позволяет использовать такой рабочий процесс:

```bash
docker build -t my-custom-image:unique-tag ./my-image-dir
kind load docker-image my-custom-image:unique-tag
kubectl apply -f my-manifest-using-my-image.yaml
```

> **ПРИМЕЧАНИЕ:** Вы можете получить список образов, присутствующих на узле кластера, с помощью `docker exec`:
> ```bash
> docker exec -it my-node-name crictl images
> ```
> где `my-node-name` — имя контейнера Docker (например, `kind-control-plane`).

> **ПРИМЕЧАНИЕ:** Политика загрузки образов в Kubernetes по умолчанию — `IfNotPresent`, если тег образа не `:latest` или не опущен (что подразумевает `:latest`), в этом случае политика по умолчанию — `Always`. `IfNotPresent` заставляет Kubelet пропускать загрузку образа, если он уже существует. Чтобы загруженные образы работали как ожидается:
> * не используйте тег `:latest`;
> * и/или укажите для ваших контейнеров `imagePullPolicy: IfNotPresent` или `imagePullPolicy: Never`.
>
> См. [политику загрузки образов Kubernetes](https://kubernetes.io/docs/concepts/containers/images/#image-pull-policy) для получения дополнительной информации.

См. также: [Использование kind с частными реестрами](https://kind.sigs.k8s.io/docs/user/private-registries/).

## Сборка образов

> **ПРИМЕЧАНИЕ:** Если вы используете Docker Desktop, обязательно прочитайте раздел [Настройки для Docker Desktop](#настройки-для-docker-desktop) сначала.

kind запускает локальный кластер Kubernetes, используя контейнеры Docker в качестве «узлов». kind использует образ узла для запуска компонентов Kubernetes, таких как kubeadm или kubelet. Образ узла, в свою очередь, собирается на основе базового образа, который устанавливает все зависимости, необходимые для работы Docker и Kubernetes в контейнере.

В настоящее время kind поддерживает один стандартный способ сборки образа узла, если у вас есть исходный код Kubernetes на вашей хост-машине (`$GOPATH/src/k8s.io/kubernetes`) – используя исходный код.

Вы также можете указать другой путь к исходному коду Kubernetes:

```bash
kind build node-image /path/to/kubernetes/source
```

> **ПРИМЕЧАНИЕ:** Сборка образов узлов Kubernetes требует всего, что требуется для сборки вышестоящего Kubernetes, мы оборачиваем сборку вышестоящего проекта. Это включает Docker с buildx. См.: https://git.k8s.io/community/contributors/devel/development.md#building-kubernetes-with-docker

Быстрый способ использовать релиз Kubernetes — указать версию напрямую, чтобы загрузить официальные tar-архивы:

```bash
kind build node-image v1.30.0
```

Если вы предпочитаете использовать уже существующие tar-архивы, например, из релиза Kubernetes, вы также можете указать их по URL или из локального каталога:

```bash
kind build node-image https://dl.k8s.io/v1.30.0/kubernetes-server-linux-arm64.tar.gz
kind build node-image $HOME/Downloads/kubernetes-server-linux-amd64.tar.gz
```

Чтобы избежать путаницы, вы можете явно указать тип сборки с помощью параметра `--type`:

```bash
kind build node-image --type url https://dl.k8s.io/v1.30.0/kubernetes-server-linux-arm64.tar.gz
kind build node-image --type file $HOME/Downloads/kubernetes-server-linux-amd64.tar.gz
kind build node-image --type release v1.30.0
kind build node-image --type source $HOME/go/src/k8s.io/kubernetes/
```

> **ПРИМЕЧАНИЕ:** Режимы, отличные от каталога с исходниками (т.е. `url`, `file` и `release`), доступны только в kind v0.24 и выше.

## Дополнительно

### Настройка вашего кластера kind

Пример конфигурационного файла kind см. в [kind-example-config](https://kind.sigs.k8s.io/docs/user/quick-start/#configuring-your-kind-cluster). Чтобы указать файл конфигурации при создании кластера, используйте флаг `--config`:

```bash
kind create cluster --config kind-example-config.yaml
```

#### Кластеры с несколькими узлами

В частности, многих пользователей могут заинтересовать кластеры с несколькими узлами. Простая конфигурация для этого может быть достигнута следующим содержимым файла:

```yaml
# Конфигурация кластера с тремя узлами (два рабочих)
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
- role: control-plane
- role: worker
- role: worker
```

#### Высокая доступность плоскости управления (HA)

Вы также можете создать кластер с несколькими узлами плоскости управления:

```yaml
# Кластер с 3 узлами плоскости управления и 3 рабочими узлами
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
- role: control-plane
- role: control-plane
- role: control-plane
- role: worker
- role: worker
- role: worker
```

#### Проброс портов на хост-машину

Вы можете пробросить дополнительные порты с узлов на хост-машину с помощью `extraPortMappings`:

```yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
- role: control-plane
  extraPortMappings:
  - containerPort: 80
    hostPort: 80
    listenAddress: "0.0.0.0" # Необязательно, по умолчанию "0.0.0.0"
    protocol: udp # Необязательно, по умолчанию tcp
```

Это может быть полезно при использовании сервисов типа NodePort или демонсетов, открывающих порты на хосте.

> **Примечание:** Привязка listenAddress к 127.0.0.1 может повлиять на возможность доступа к сервису.

Возможно, вам также будет интересно ознакомиться с [Руководством по Ingress](https://kind.sigs.k8s.io/docs/user/ingress/) и [Руководством по LoadBalancer](https://kind.sigs.k8s.io/docs/user/loadbalancer/).

#### Установка версии Kubernetes

Вы также можете указать конкретную версию Kubernetes, задав образ контейнера узла. Доступные теги образов можно найти на [странице релизов](https://github.com/kubernetes-sigs/kind/releases). Пожалуйста, используйте SHA256-сумму для вашей желаемой версии Kubernetes, как показано в примере:

```yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
- role: control-plane
  image: kindest/node:v1.16.4@sha256:b91a2c2317a000f3a783489dfb755064177dbc3a0b2f4147d50f04825d016f55
- role: worker
  image: kindest/node:v1.16.4@sha256:b91a2c2317a000f3a783489dfb755064177dbc3a0b2f4147d50f04825d016f55
```

#### Включение шлюзов функций (Feature Gates) в вашем кластере

Feature gates — это набор пар `key=value`, описывающих альфа- или экспериментальные возможности. Чтобы включить шлюз, вам нужно настроить конфигурацию kubeadm, и это зависит от того, какой шлюз и компонент вы хотите включить. Пример конфигурации kind:

```yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
featureGates:
  FeatureGateName: true
```

(В примере приведено дважды, возможно, опечатка в оригинале, но мы оставим как есть.)

#### Настройка kind для использования прокси

Если вы запускаете kind в среде, требующей прокси, вам может потребоваться настроить kind для его использования.

Вы можете настроить kind на использование прокси с помощью одной или нескольких следующих переменных окружения (регистр имеет значение, верхний регистр имеет приоритет):

* `HTTP_PROXY` или `http_proxy`
* `HTTPS_PROXY` или `https_proxy`
* `NO_PROXY` или `no_proxy`

> **ПРИМЕЧАНИЕ:** Если вы установите прокси, он будет передан во все узлы kind. kind автоматически добавит некоторые адреса в `NO_PROXY` перед передачей его узлам, чтобы компоненты Kubernetes подключались друг к другу напрямую, но вам может потребоваться настроить дополнительные адреса в зависимости от вашего использования.

#### Экспорт логов кластера

kind позволяет экспортировать все логи, связанные с кластером, для изучения. Чтобы экспортировать логи из кластера по умолчанию (имя контекста `kind`):

```bash
kind export logs
Exported logs to: /tmp/396758314
```

Как и все другие команды, если вы хотите выполнить действие с кластером с другим именем контекста, используйте флаг `--name`.

Как видите, kind поместил все логи для кластера `kind` во временный каталог. Если вы хотите указать местоположение, просто добавьте путь к каталогу после команды:

```bash
kind export logs ./somedir
Exported logs to: ./somedir
```

Структура логов будет выглядеть примерно так:

```
.
├── docker-info.txt
└── kind-control-plane/
    ├── containers
    ├── docker.log
    ├── inspect.json
    ├── journal.log
    ├── kubelet.log
    ├── kubernetes-version.txt
    └── pods/
```

Логи содержат информацию о хосте Docker, контейнерах, запускающих kind, самом кластере Kubernetes и т.д.
