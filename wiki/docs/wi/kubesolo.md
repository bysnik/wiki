## Сборка KubeSolo из исходных кодов

### Требования

- **Go** версии 1.24 или новее
- **Docker** (для ARM-сборок и управления зависимостями)
- **Наборы инструментов для кросс-компиляции** (опционально, для сборки под другие архитектуры)

### Установка инструментов кросс-компиляции

Для сборки под несколько архитектур установите необходимые кросс-компиляторы:

```bash
sudo make install-cross-compilers
```

Будут установлены:
- `gcc-aarch64-linux-gnu` (для ARM64)
- `gcc-x86-64-linux-gnu` (для AMD64)
- `gcc-arm-linux-gnueabihf` (для ARM/ARMHF)
- `riscv64-linux-gnu-gcc` (для RISCV64)

### Базовая сборка

Сборка для вашей текущей платформы (результат — `./dist/kubesolo`):

```bash
make build
```

### Кросс-платформенная сборка

Сборка для конкретной архитектуры с помощью переменной `GOARCH`:

```bash
# ARM64
make build GOARCH=arm64

# AMD64
make build GOARCH=amd64

# ARM (ARMHF)
make build GOARCH=arm

# RISCV64
make build GOARCH=riscv64
```

### Сборка для Alpine Linux / musl

Для совместимости с Alpine Linux соберите статически слинкованные бинарники, совместимые с musl:

```bash
# Установка musl-кросс-компиляторов (однократно)
sudo make install-musl-cross-compilers

# Сборка musl-бинарника под конкретную архитектуру
make build-musl GOARCH=amd64
make build-musl GOARCH=arm64

# Сборка всех поддерживаемых musl-архитектур
make build-all-musl
```

> **Примечание:** musl-сборки в настоящее время поддерживаются только для архитектур `amd64` и `arm64`.

### Указание пути для выходного файла

Переопределите путь с помощью переменной `OUTPUT`:

```bash
# С произвольным именем
make build OUTPUT=./kubesolo-custom

# С именем, отражающим платформу
make build GOARCH=arm OUTPUT=./dist/kubesolo-arm

# В другую директорию
make build OUTPUT=./bin/kubesolo
```

### Комбинированные примеры

```bash
# ARM-бинарник с пользовательским именем
make build GOARCH=arm OUTPUT=./dist/kubesolo-linux-arm

# AMD64-бинарник для CI/CD
make build GOARCH=amd64 OUTPUT=./artifacts/kubesolo-linux-amd64
```

### Разработка и тестирование

Для разработки и тестирования можно запускать KubeSolo напрямую без сборки:

```bash
# Загрузка зависимостей (только один раз)
make deps

# Запуск в режиме разработки
make dev
```

### Важные замечания

- **ARM-сборки:** для архитектуры ARM бинарники containerd собираются с использованием кросс-компиляции через Docker для достижения наилучшей совместимости.
- **Зависимости:** процесс сборки автоматически загружает необходимые зависимости (containerd, runc, CNI-плагины) для целевой архитектуры.
- **CGO:** все сборки используют CGO для повышения производительности и совместимости с системными библиотеками.

## KubeSolo: лёгкий одноузловой Kubernetes

KubeSolo — это полноценный Kubernetes, работающий в одном процессе и потребляющий менее 200 МБ оперативной памяти. Он создан для сред, где раньше использовали Docker или Podman: пограничные устройства, промышленные шлюзы, разработческие ноутбуки, удалённые объекты, IoT-контроллеры. Вместо того чтобы отключать кластерную логику в существующем дистрибутиве, мы полностью удалили весь код, связанный с многозвенностью, — и получили совместимый с API Kubernetes стек без избыточных накладных расходов.

## Что это такое?

KubeSolo — это Kubernetes, из которого удалены все компоненты, имеющие смысл только в кластере из нескольких узлов:

- логика кворума etcd;
- выборы лидера;
- оверлейные сети для многозвенной связи;
- распределение управляющей плоскости.

Всё это не просто отключено, а полностью исключено из кодовой базы.

Оставшееся — это полноценный цикл управления Kubernetes в едином процессе: API-сервер, контроллер-менеджер и kubelet работают совместно. Ваши существующие манифесты, Helm-чарты и CRD работают без изменений.

**Целевые сценарии** — любые хосты, где раньше использовался Docker или Podman:
- пограничные устройства;
- заводское оборудование;
- ноутбуки разработчиков;
- удалённые площадки;
- IoT-шлюзы;
- киоски и терминалы.

Те же OCI-образы, но с полноценной экосистемой Kubernetes.

В состав KubeSolo входят:

- **containerd** и **crun** — контейнерный рантайм;
- **CoreDNS** — разрешение DNS-имён;
- **Kine** — хранилище на основе SQLite (вместо etcd).

Поставляется как единый бинарный файл с минимальными зависимостями от ОС (ядро и смонтированные cgroups), безопасными настройками по умолчанию и всеми необходимыми компонентами для работы в изолированных средах.

## Почему он такой лёгкий?

Потребление памяти менее 200 МБ достигается за счёт того, что кластерный код не просто «спит», а полностью отсутствует. Большинство облегчённых дистрибутивов Kubernetes лишь урезают стандартную сборку, оставляя многозвенный код неактивным. KubeSolo идёт с другой стороны: удаляется всё, что требует более одного узла.

Три ключевых решения:

1. **Вместо etcd** — SQLite через Kine.
2. **Вместо штатного планировщика** — лёгкий вебхук `NodeSetter`, который обрабатывает планирование на одном узле без полной логики планировщика.
3. **Все компоненты** — в одном процессе, а не в отдельных бинарных файлах.

В итоге полноценный Kubernetes работает на устройствах с 512 МБ ОЗУ, на флеш-накопителях и в изолированных сетях.

## Почему размер бинарного файла может быть больше, чем у других дистрибутивов?

KubeSolo поставляется в двух вариантах, чтобы соответствовать разным средам развёртывания:

| Вариант | Размер бинарника | Требуется интернет | Когда использовать |
|---------|------------------|-------------------|-------------------|
| **Онлайн** (по умолчанию) | Меньше | Да | Устройства с постоянным доступом в интернет, где важен размер файла |
| **Офлайн** | Больше | Нет | Изолированные среды, заводские цеха, устройства с нестабильным или отсутствующим интернетом |

В офлайн-варианте все необходимые образы контейнеров, CNI-плагины и зависимости упакованы прямо в бинарник — ничего не скачивается при установке или запуске. Онлайн-вариант подтягивает образы из публичных реестров при старте, сохраняя бинарник компактным.

## Флаги командной строки

KubeSolo поддерживает следующие параметры запуска (можно задавать через переменные окружения):

| Флаг | Переменная окружения | Описание | По умолчанию |
|------|-----------------------|----------|-------------|
| `--version` | – | Показать версию и выйти | – |
| `--path` | `KUBESOLO_PATH` | Путь к каталогу с конфигурационными файлами KubeSolo | `/var/lib/kubesolo` |
| `--apiserver-extra-sans` | `KUBESOLO_APISERVER_EXTRA_SANS` | Список дополнительных имён (SAN) для TLS-сертификата API-сервера (IP или DNS, через запятую) | `""` |
| `--portainer-edge-id` | `KUBESOLO_PORTAINER_EDGE_ID` | Идентификатор Edge для Portainer | `""` |
| `--portainer-edge-key` | `KUBESOLO_PORTAINER_EDGE_KEY` | Ключ Edge для Portainer | `""` |
| `--portainer-edge-async` | `KUBESOLO_PORTAINER_EDGE_ASYNC` | Включить асинхронный режим Edge Portainer | `false` |
| `--load-balancer` | `KUBESOLO_LOAD_BALANCER` | Включить балансировщик нагрузки; тогда для новых сервисов типа LoadBalancer будет подставляться IP узла | `true` |
| `--local-storage` | `KUBESOLO_LOCAL_STORAGE` | Включить локальное хранилище | `true` |
| `--local-storage-shared-path` | `KUBESOLO_LOCAL_STORAGE_SHARED_PATH` | Путь к общей файловой системе для локального хранилища | `""` |
| `--debug` | `KUBESOLO_DEBUG` | Включить отладочное логирование | `false` |
| `--pprof-server` | `KUBESOLO_PPROF_SERVER` | Включить pprof-сервер для профилирования | `false` |
| `--full` | `KUBESOLO_FULL` | Устаревший (не влияет). Оставлен для обратной совместимости | `false` |
| `--container-mode` | `KUBESOLO_CONTAINER_MODE` | Запуск внутри контейнера (автоопределяется). Настраивает cgroups, монтирования, DNS, пороги вытеснения | _авто_ |
| `--db-wal-repair` | `KUBESOLO_DB_WAL_REPAIR` | Выполнять проверку целостности SQLite при старте и восстанавливать артефакты WAL/SHM при обнаружении повреждений | `false` |
| `--disable-ipv6` | `KUBESOLO_DISABLE_IPV6` | Отключить IPv6 для обратных зон CoreDNS и регистрации адреса узла kubelet | `false` |
| `--startup-timeout` | `KUBESOLO_STARTUP_TIMEOUT` | Максимальное время (сек) ожидания готовности каждого компонента при старте. Увеличьте для медленных накопителей (SD-карты) | `600` |
| `--d2k` | `KUBESOLO_D2K` | Встроить [d2k](https://github.com/portainer/d2k) и открыть совместимый с Docker API endpoint по mTLS на порту 2376 | `false` |
| `--d2k-namespace` | `KUBESOLO_D2K_NAMESPACE` | Пространство имён, в которое разворачивается d2k и через которое транслируются Docker-вызовы (действует только при `--d2k`) | `d2k` |

## kubesoloctl — утилита управления

Download the binary for your platform from the [releases page](https://github.com/portainer/kubesolo/releases) (`kubesoloctl-<os>-<arch>`), put it on your PATH, then:

`kubesoloctl` — это единый автономный бинарный файл для управления жизненным циклом KubeSolo: установка, обновление, сброс, удаление, настройка kubeconfig, а также **режим контейнера** для запуска KubeSolo на macOS, Windows (WSL2) или любом Linux-хосте с контейнерным движком.

Он работает в двух режимах:

- **Хост-режим** (Linux): KubeSolo запускается как системный сервис (systemd, OpenRC, SysV и т.д.). Это production-путь для пограничных устройств.
- **Режим контейнера**: KubeSolo работает внутри контейнера. Используется для **разработки и CI** на macOS, WSL2 или любом Linux-хосте с Docker/Podman. На macOS этот режим единственный, так как бинарник KubeSolo предназначен для Linux.

### Команды

| Команда | Назначение |
|---------|-----------|
| `install` | Установить KubeSolo и настроить сервис / запустить контейнер |
| `check` | Выполнить только предустановочные проверки |
| `upgrade --version=<v>` | Обновить до указанной версии с сохранением конфигурации |
| `reset` | Очистить всё состояние кластера и начать заново (установка сохраняется) |
| `uninstall` | Остановить и удалить KubeSolo, файлы сервисов и (опционально) данные |
| `kubeconfig` | Вывести, записать, извлечь или просмотреть административный kubeconfig |
| `d2k fetch` | Настроить Docker-контекст для d2k-эндпоинта (в режиме контейнера) |
| `download` | Собрать офлайн-пакет для установки на изолированной машине |
| `version` | Показать информацию о версии |

### Режимы работы

| Режим | Флаг | Где выполняется | Типичное использование |
|-------|------|-----------------|------------------------|
| `service` | _(по умолчанию на Linux)_ | Бинарник KubeSolo как системный сервис | Production / пограничные устройства |
| `container` | `--run-mode=container` | KubeSolo в контейнере | Разработка / CI (macOS, WSL2, Linux) |
| `daemon` | `--run-mode=daemon` | Фоновый процесс без системы инициализации | Минималистичные хосты |
| `foreground` | `--run-mode=foreground` | Выполняется в терминале | Отладка |

На **macOS** режим автоматически принудительно устанавливается в `container`.

### Основные флаги для `install`

| Флаг | Переменная окружения | По умолчанию | Описание |
|------|-----------------------|-------------|----------|
| `--version` | `KUBESOLO_VERSION` | соответствует версии CLI | Версия KubeSolo для установки |
| `--path` | `KUBESOLO_PATH` | `/var/lib/kubesolo` | Каталог данных |
| `--run-mode` | `KUBESOLO_RUN_MODE` | `service` | `service`, `daemon`, `foreground` или `container` |
| `--name` | `KUBESOLO_NAME` | `kubesolo` | Имя экземпляра (имя контейнера + контекст kubeconfig) |
| `--image` | `KUBESOLO_IMAGE` | `portainer/kubesolo:<version>` | Образ контейнера (в режиме контейнера) |
| `--container-ports` | `KUBESOLO_CONTAINER_PORTS` | _(нет)_ | Проброс портов для рабочих нагрузок (режим контейнера) |
| `--d2k` | `KUBESOLO_D2K` | `false` | Включить Docker-совместимый API-транслятор |
| `--local-storage` | `KUBESOLO_LOCAL_STORAGE` | `false` | Включить провизионер локального хранилища |
| `--offline-install` | `KUBESOLO_OFFLINE_INSTALL` | _(нет)_ | Установка из локального архива/бинарника вместо скачивания |
| `--proxy` | `KUBESOLO_PROXY` | _(нет)_ | HTTP/HTTPS-прокси, добавляемый в окружение сервиса |

### Подробности режима контейнера

В режиме контейнера KubeSolo работает в собственном сетевом пространстве. Сервисы типа NodePort/LoadBalancer или `hostPort` будут доступны только если соответствующие порты опубликованы на хосте — аналогично механизму `extraPortMappings` в Kind. Для публикации портов используется флаг `--container-ports`:

```bash
kubesoloctl install --run-mode=container --container-ports=9001,8080:80,9000-9100,53/udp
```

Синтаксис записей:

| Запись | Значение |
|--------|----------|
| `9001` | хост `9001` → контейнер `9001` |
| `9000-9100` | диапазон с совпадением портов (1:1) |
| `8080:80` | хост `8080` → контейнер `80` |
| `127.0.0.1:80:80` | привязка только к конкретному IP хоста |
| `53/udp` | UDP (по умолчанию подразумевается TCP) |

Одиночные порты и диапазоны привязываются ко всем интерфейсам (доступны с других машин). Чтобы ограничить их localhost, используйте форму `127.0.0.1:хост:контейнер`. Опубликованные порты сохраняются при обновлении и сбросе.

### d2k (Docker-совместимый API)

Если KubeSolo установлен с флагом `--d2k`, можно создать локальный Docker-контекст, который обращается к встроенному транслятору через mTLS:

```bash
kubesoloctl d2k fetch                 # для экземпляра по умолчанию
kubesoloctl d2k fetch --name prod     # для именованного экземпляра

docker --context kubesolo ps          # использование
```

Команда `d2k fetch` копирует CA и клиентский сертификат из контейнера в `~/.docker/d2k/<имя>/` и создаёт (или обновляет) Docker-контекст, указывающий на опубликованный порт d2k. Поскольку порт назначается случайно, повторно выполняйте `d2k fetch` после каждого обновления или сброса.

### Несколько экземпляров

С помощью `--name` можно запускать несколько независимых экземпляров KubeSolo в режиме контейнера. Каждый получает свой контейнер (`kubesolo-<имя>`), том данных, контекст kubeconfig и (при включённом d2k) Docker-контекст. Указывайте тот же `--name` для команд `upgrade`, `reset`, `uninstall`, `kubeconfig fetch` и `d2k fetch`, чтобы воздействовать на нужный экземпляр.

### Офлайн / изолированная среда

Команда `download` создаёт автономный набор файлов (архив с релизом KubeSolo и сам бинарник `kubesoloctl`) для переноса на машину без доступа в интернет:

```bash
# На машине с интернетом (укажите --arch, если целевая архитектура отличается):
kubesoloctl download --version=v1.1.8 --path=./bundle --arch=arm64

# На целевой машине:
sudo ./kubesoloctl install --offline-install=./bundle/kubesolo-v1.1.8-linux-arm64.tar.gz
```
